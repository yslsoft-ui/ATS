# 0010. 자동 상장 및 상장폐지 감지/스케줄링 아키텍처 도입

* Status: accepted
* Date: 2026-06-19

## Context and Problem Statement

시스템에 신규 종목이 상장되거나 기존 종목이 상장폐지될 때, 기존에는 관리자가 웹 UI에서 수동으로 "거래소 자산 동기화" 버튼을 클릭하여 수집 대상을 업데이트해야 했습니다. 이로 인해 다음과 같은 문제점이 존재했습니다:
1. **실시간 대응 부재**: 신규 상장이나 상장폐지가 발생했을 때 관리자가 즉각 개입하지 않으면 실시간 틱 수집 및 자동 매매 엔진이 누락되는 공백이 생깁니다.
2. **사전 대응 한계**: 거래소는 상장 및 상장폐지 일정을 수시간 혹은 수일 전에 공지사항으로 사전 예고하지만, 기존 시스템은 실제 상장된 이후에만 종목을 인지하고 백필(Backfill)을 실행할 수 있어 사전에 수집 준비 및 웜업(Warm-up)을 처리할 수 없었습니다.
3. **API 차단 제약**: 업비트의 경우 Cloudflare 보안 장벽으로 인해 백엔드 서버에서 공지사항 페이지 크롤링이 차단(HTTP 403 / Hang)되어 안정적인 공지사항 조회에 기술적 한계가 존재합니다.

## Decision Outcome

수동 개입을 배제하고 상장/상폐 이벤트에 선제적·안정적으로 대응하기 위해 **주기적 공지 폴링 및 30분 전 선제 기동 방식의 자동 스케줄러**를 도입하기로 결정하였습니다.

### 세부 구현 내용

1. **상장/상장폐지 예정 이벤트 테이블 (`planned_asset_events`) 신설**:
   * 예정된 이벤트를 영속적으로 저장하고 중복 감지를 방지하기 위한 전용 데이터베이스 테이블을 설계합니다.
   * 상태 필드(`status` - `'PLANNED'`, `'EXECUTED'`, `'CANCELLED'`)를 도입하여 상태 전이를 추적합니다.

2. **거래소별 이원화 감지 파이프라인 구축**:
   * **빗썸 (Bithumb)**: 
     * 공식 공지사항 API(`https://api.bithumb.com/v1/notices`)를 12시간 주기로 폴링합니다.
     * 카테고리가 `'신규상장'` 또는 `'거래지원종료'`인 신규 공지를 발견하면 본문 HTML(`pc_url`) 내의 `__NEXT_DATA__` 상태값을 파싱하여 대상 종목 코드(Ticker)와 예정 일시(상장/상폐 시간)를 정규식으로 자동 추출합니다.
     * 최초 인지 시점에 ZMQ IPC를 통해 **웹 UI에 즉시 토스트 팝업 알림** 및 `ASSET_LISTING_SCHEDULED` / `ASSET_DELISTING_SCHEDULED` **시스템 이벤트**를 발행하고 DB에 `PLANNED` 상태로 등록합니다.
   * **업비트 (Upbit)**:
     * 웹 서버 크롤링 차단 문제를 해결하기 위해, 공식 API인 `https://api.upbit.com/v1/market/all`을 1시간 주기로 폴링합니다.
     * 신규 종목이 마켓 목록에 새롭게 감지되는 즉시 사전 예정 시간 없이 **실시간 상장 등록 및 즉시 수집** 프로세스로 연동합니다.
   * **KIS (한국투자증권)**:
     * 1일 1회 마스터 파일(`kospi_code.mst.zip`, `kosdaq_code.mst.zip`)을 자동 다운로드하여 비교합니다.
     * 신규 종목 감지 시 시스템 과부하를 막기 위해 `is_active=0` (수집 비활성) 상태로 등록하되, **"KIS 신규 상장 감지" 토스트 알림 및 이벤트**를 띄워 관리자가 선택적으로 기용하도록 돕습니다. 상장폐지 시 즉시 비활성화(`is_active=0`, `is_delisted=1`)합니다.

3. **예정 시간 30분 전 선제 자동 수집 기동**:
   * `CollectorService` 내 백그라운드 스케줄러가 1분마다 `planned_asset_events` 중 실행 예정 시각 30분 전(`scheduled_at - 1800초`)에 도달한 이벤트를 조회합니다.
   * 해당 시점에 `exchange_assets` 테이블의 `is_active` 값을 1(또는 상폐 시 0)로 변경하고, ZMQ `update_symbols` 제어 신호를 보내 즉시 웹소켓 실시간 구독 및 백필을 백그라운드로 작동시켜 거래 시작 시점까지 엔진 웜업을 완료할 수 있도록 합니다.

4. **실제 실행 시점의 완료 처리 및 알림**:
   * 실제 예정 시간에 도달하여 첫 실시간 틱 데이터가 수집되거나 동기화가 확정되면, 해당 이벤트 상태를 `EXECUTED`로 마크하고 **"상장/상장폐지 완료" 시스템 이벤트 및 최종 토스트 알림**을 발송합니다.

## Consequences

* **Positive**:
  * 관리자가 직접 자산 동기화 버튼을 누르지 않아도 상장/상폐 이벤트에 기민하고 정밀하게 대처할 수 있어 시세 누락 공백이 해소됩니다.
  * 거래 시작 30분 전 선제 백필 및 웹소켓 구독을 보장하여 거래 시작 시점에 지표 계산이 완전히 준비된 상태(Warm-up 완료)로 자동 매매를 시작할 수 있습니다.
  * 비공식 크롤링 한계를 고려하여 거래소별 최적의 API(빗썸 공지사항, 업비트 마켓목록, KIS 마스터파일)로 이원화 설계하여 안정성을 극대화했습니다.
* **Negative**:
  * 공지사항 제목 및 본문 텍스트 내에서 날짜와 심볼을 정규식으로 추출하므로, 거래소 측에서 공지 본문 작성 포맷을 대폭 변경할 경우 파싱 오작동이 일어날 가능성이 있어 정밀한 예외 처리가 요구됩니다.
