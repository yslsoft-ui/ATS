# 0003. PortfolioManager 도입을 통한 자산 관리 및 주문 실행 분리

* Status: accepted
* Date: 2026-05-15

## Context and Problem Statement

기존 시스템에서는 전략에서 발생한 매매 신호(`TradeSignal`)가 단순히 로그를 출력하거나 알림을 보내는 수준에 그쳤습니다. 실제 자산(가상 또는 실제)의 상태를 관리하고, 체결 엔진과 연동하여 잔고를 업데이트하는 체계적인 구조가 부족했습니다. 또한, 실시간 가상 매매와 실제 매매 로직을 유연하게 교체할 수 있는 추상화 레이어가 필요했습니다.

## Decision Outcome

자산 상태 관리와 주문 실행을 전담하는 `PortfolioManager` 레이어를 도입하기로 결정했습니다.

### 세부 내용

1. **Portfolio & Position**: 개별 계좌의 현금 잔고와 종목별 보유 수량, 평균 단가를 객체 지향적으로 관리합니다.
2. **OrderExecutor 인터페이스**: 주문 체결 방식을 추상화하여, `VirtualExecutor`(가상 체결)와 추후 구현될 `UpbitExecutor`(실제 API 체결)를 동일한 인터페이스로 다룰 수 있게 합니다.
3. **Simulation Fallback**: 호가창(`Orderbook`) 데이터가 없는 경우에도 현재 체결가(`trade_price`)를 기반으로 즉시 체결을 지원하여 데이터 누락 시에도 시스템의 연속성을 보장합니다.
4. **DB 연동**: 포트폴리오 상태와 상세 주문 이력을 DB에 저장하여 세션이 종료되어도 자산 상태를 복구할 수 있도록 설계했습니다. (초기 단계에서는 메모리 관리 후 주기적 저장 방식 채택)

## Consequences

* **Positive**: 
    - 전략 로직(`TradeEngine`)과 자산 관리 로직(`PortfolioManager`)이 명확히 분리되어 유지보수성이 향상됩니다.
    - 동일한 전략으로 여러 개의 가상 계좌를 운영하거나, 가상/실제 매매를 동시에 테스트하기 용이해집니다.
* **Negative**: 시스템 전체의 복잡도가 다소 증가하며, 실시간 체결 처리 시 오버헤드가 발생할 수 있습니다.
* **Neutral**: 모든 매매 행위가 `PortfolioManager`를 거치게 되므로, 강력한 리스크 관리 기능을 추가할 수 있는 기반이 마련되었습니다.
