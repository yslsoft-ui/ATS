# 0009. 시장 데이터 클린업 데몬 제어 아키텍처 및 원격 인터페이스 도입

* Status: accepted
* Date: 2026-06-13

## Context and Problem Statement

시스템이 실시간 체결 데이터(Ticks)와 분봉 데이터(Candles)를 지속적으로 수집하면서 SQLite 데이터베이스의 크기가 급격히 팽창하는 문제가 발생했습니다. 백그라운드에서 주기적으로 오래된 데이터를 보존 정책(TTL)에 따라 소거하는 클린업 데몬(`market_cleanup_daemon.py`)이 도입되었으나, 다음과 같은 제어 및 관리적 보완이 요구되었습니다:
1. **데몬 상태 실시간 모니터링**: 웹 UI 대시보드에서 클린업 데몬의 활성/일시정지 상태 및 데이터 삭제량 통계를 실시간으로 추적할 수 있어야 합니다.
2. **원격 수동 제어판 및 동시성 보호**: 사용자가 특정 날짜 기준 데이터를 임의로 즉시 청소하거나 미리보기(Preview) 할 수 있어야 하나, 자동 스케줄러 기동 도중 수동 삭제가 중복 실행되어 SQLite DB의 동시성 락 경합이나 데이터 무결성이 손상될 리스크가 큽니다.
3. **지연 최소화와 반응성**: 사용자가 자동 정리를 정지(`PAUSED`)하거나 다시 시작(`ACTIVE`)할 때, 긴 시간 동안 sleep 하던 스케줄러 루프가 즉각 제어 신호에 반응해야 합니다.

## Decision Outcome

클린업 데몬의 라이프사이클을 통제하고 비동기식 명령-응답(Command-Response) 구조 및 강력한 뮤텍스를 확보하기 위해 다음 아키텍처를 도입하기로 결정했습니다.

### 세부 내용

1. **원격 명령 수동 삭제/조회의 데몬 단일화**:
   - API 서버가 데이터베이스를 직접 읽거나 삭제하지 않고, ZMQ 제어 채널(`market_cleanup_control`)을 통해 데몬에 요청을 전송하며, 실제 모든 DB 조회/삭제 기동 주체는 오직 `market_cleanup_daemon.py`로만 단일화합니다.

2. **asyncio.Event 기반의 스케줄러 지연 없는 활성/정지 제어**:
   - `PAUSED` 상태일 때는 `asyncio.Event`를 사용해 무기한 대기하다가, `cleanup_start` 신호가 들어오는 순간 즉각 스케줄러를 깨우며, 대기 중일 때는 `asyncio.wait_for` 패턴을 사용해 대기 타임아웃 윈도우 계산 도중이라도 즉시 인터럽트하여 기동할 수 있는 반응성을 확보합니다.

3. **asyncio.Lock 기반의 2중 동시성 방어 (Mutex Guard)**:
   - 자동 정리 스케줄러 루프와 수동 즉시 정리(`cleanup_run_once`), 미리보기(`cleanup_preview`) 등 DB에 진입하는 모든 행위에 데몬 전역 뮤텍스 락(`self._cleanup_lock = asyncio.Lock()`)을 통과하도록 강제합니다.
   - 이미 락이 확보된 상태에서 추가적인 조회/삭제 명령이 올 경우 데몬 단에서 즉각 기각 처리하고 실패 패킷을 송출해 락 경합을 원천 차단합니다.
   - 프론트엔드에서도 데몬 상태가 `RUNNING_ONCE`이거나 동작 중인 경우 버튼을 `disabled` 처리하여 2중으로 물리적 중복 입력을 차단합니다.

4. **비동기 command_id 기반의 분할 타임아웃 ACK 보장**:
   - 제어 명령 발행 시 요청 본문에 고유 `command_id`를 명시하며, 백엔드는 즉시 `pending` 상태로 응답한 후 실제 연산이 끝나는 시점에 ZMQ `cleanup_signal` 토픽을 통해 웹소켓으로 결과 패킷을 브로드캐스트합니다.
   - 명령의 무거운 정도에 따라 프론트엔드 타임아웃 규칙을 차등 적용합니다:
     - 라이프사이클 명령 (`cleanup_start`, `cleanup_stop`, `cleanup_restart_daemon`): **3초**
     - 조회 명령 (`cleanup_preview`): **5초**
     - 삭제 실행 명령 (`cleanup_run_once`): **30초**

## Consequences

* **Positive**:
  - 데이터 삭제 작업이 한 곳(데몬)으로 단일화되어 비즈니스 일관성과 보안성이 극대화되었습니다.
  - 전역 락과 버튼 비활성화를 통한 2중 가드로 SQLite 락 경합으로 인한 데이터베이스 커럽션 리스크를 배제하였습니다.
  - 실시간 이벤트와 텔레메트리 캐시 처리로 웹서버와 DB에 주는 부하를 최소화하였습니다.
* **Negative**:
  - 비동기 웹소켓 ACK 응답 매칭을 관리해야 하므로 프론트엔드 통신 상태 코드 복잡성이 소폭 증가합니다.
